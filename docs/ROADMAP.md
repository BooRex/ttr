# Roadmap (детальный)

Этот план сфокусирован на стабильности, чистоте кода и ускорении дальнейшей разработки.

## Приоритеты

- **P0**: стандарты кода + quality gates, единый источник правды для карты, базовая реструктуризация frontend.
- **P1**: mobile-first UX и декомпозиция крупных UI-модулей.
- **P2**: анимации и визуальный polish без просадки производительности.

## Quick Wins (1-2 дня)

1. Подключить `ESLint` + `typescript-eslint` + `react-hooks` и добавить скрипты `lint` / `lint:fix`.
2. Добавить `format:check` (Prettier/Biome) и единый форматтер в workflow.
3. Ввести `typecheck + lint + test` как обязательный CI-gate для PR.
4. Добавить тест консистентности карты: `shared gameplay cities === layout cities`.
5. Убрать хардкод `DEFAULT_EUROPE_CITIES` в `calibrator` и читать данные из shared-источника.

## Фаза 0 — Baseline и стандарты (P0, 8-14 ч)

### Задачи

- Добавить линтинг и форматирование для `apps/client`, `apps/server`, `packages/shared`.
- Добавить скрипты в корневой `package.json`: `lint`, `typecheck`, `format:check`.
- Поднять CI-job с fail-fast на `typecheck/lint/test`.
- Зафиксировать правила в `docs/GETTING_STARTED.md`.

### Результат

- Единый стиль кода, меньше “шумных” PR, раннее обнаружение регрессий.

### Риски

- Большой стартовый diff форматирования.
- Временный рост количества warning/error после включения ESLint.

### DoD

- Любой PR падает при нарушении `lint/typecheck/test`.
- Локально одной командой можно прогнать весь quality-check.

## Фаза 1 — Single Source of Truth для карты (P0, 16-28 ч)

### Проблема сейчас

Карта и координаты частично дублируются между `shared`, `client` и `calibrator`.

### Целевая схема

- `packages/shared/src/maps/europe.gameplay.ts`
  - `cities`, `routes`, `destinationDeck`.
- `packages/shared/src/maps/europe.layout.ts`
  - `europeCities` (`x/y + label.dx/dy`), `board`, `background`.
- `client` импортирует layout только из `@ttr/shared` (через адаптер).
- `calibrator` читает/сохраняет этот же layout-источник.

### Задачи

- Вынести layout-данные в `shared` как канонический источник.
- Удалить локальные дубли в `client`/`calibrator`.
- Добавить в shared тест консистентности городов gameplay/layout.

### Результат

- Одно изменение координат автоматически валидно для всех потребителей.

### Риски

- Миграция формата данных в `calibrator`.
- Нужна аккуратная обратная совместимость по именам городов.

### DoD

- Нет дублей `europeCities` в нескольких местах.
- Проходят тесты консистентности карты.

## Фаза 2 — Упрощение структуры frontend (P0, 24-36 ч)

### Цель

Сделать структуру проще и читаемее, убрать странный перекос с одиночной feature (`event-log`).

### Предложенная структура (FSD-lite, без перегруза)

- `app/` — entry, providers, store wiring, socket wiring.
- `pages/` — `LobbyPage`, `WaitingRoomPage`, `GamePage`, `ResultsPage`.
- `widgets/` — большие UI-блоки (`GameBoard`, `Topbar`, `SidePanels`).
- `features/` — сценарии пользователя (`claim-route`, `draw-cards`, `draw-destinations`, `station-build`, `event-log`, `language-switch`).
- `entities/` — доменные модели/утилиты (`game`, `route`, `player`, `destination`).
- `shared/` — ui-kit, lib, config, helpers.

### Задачи

- Переместить `event-log` в единую систему feature-модулей.
- Выделить еще 3-5 features (route claim, draw actions, station build и т.д.).
- Сделать `App` orchestration-only, без тяжелой бизнес-логики.

### Результат

- Проект легче читать и развивать, меньше когнитивной нагрузки.

### DoD

- `App` и `GameScreen` не содержат “толстую” доменную логику.
- Основные сценарии разнесены по feature-модулям.

## Фаза 3 — Mobile-first UX (P1, 24-34 ч)

### Цель

Сделать игру реально удобной на телефонах, включая портретный режим.

### Задачи

- Перейти на mobile-first стили, desktop через `min-width` enhancement.
- Свести критичные действия к крупным touch-target.
- Для portrait: упростить layout (карта + actions dock + быстрый доступ к картам/событиям).
- Опционально: feature-flag `compactMobileUI` для отдельного мобильного варианта.

### Результат

- Полный ход игрока можно сделать на 360-430px ширине без блокирующих костылей.

### Риски

- Конфликты pan/zoom карты с touch UI.
- Меньше полезной площади под карту.

### DoD

- Все базовые действия доступны в portrait и landscape.
- Нет сценариев, где пользователь “застрял” на мобилке.

## Фаза 4 — Декомпозиция крупных компонентов (P1, 24-40 ч)

### Задачи

- Разбить `BoardCanvas` на подмодули:
  - геометрия,
  - рендер маршрутов,
  - рендер городов/лейблов,
  - жесты/масштаб,
  - анимации.
- Разбить `ActionPanel` на независимые блоки:
  - main actions,
  - pending choices,
  - claim route options,
  - waiting state.
- Вынести математику/алгоритмы в чистые функции с unit-тестами.

### Результат

- Меньше file-size и проще точечные изменения без побочных эффектов.

### DoD

- Крупные файлы >400 LOC разбиты на модули.
- Критическая логика покрыта тестами.

## Фаза 5 — Casual-анимации и визуальный polish (P2, 16-28 ч)

### Задачи

- Анимация добора карт (из колоды/рынка в руку).
- Анимация подтверждения захвата маршрута и постройки станции.
- Микро-анимации в событиях и смене хода.
- Поддержка `prefers-reduced-motion`.

### Результат

- Более живой и казуальный UX без ущерба играбельности.

### Риски

- Просадки FPS на слабых девайсах.
- Сложность синхронизации анимации с realtime-обновлениями.

### DoD

- Анимации не блокируют взаимодействие.
- Нет визуальных лагов на массовых обновлениях состояния.
